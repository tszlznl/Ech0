---
title: 向量检索（语义搜索）
description: 给历史 Echo 建向量索引，让 AI 问答按「意思」检索；含维度对齐排坑
---

**向量检索** 给你的历史 Echo 建一份向量索引，让 [AI 问答](/docs/guide/chat) 能按**语义**而不是字面去找内容。

::: tip 可选功能
不配也能正常对话。区别在于检索质量：
- **不开**：检索退化为**关键词子串匹配**——换个说法（如查「心情低落」却写的是「很沮丧」）就可能搜不到。
- **开启**：**语义召回**——意思相近就能命中，更贴合「我以前是不是写过关于 X 的想法」这类回忆式提问。
:::

配置在 **系统设置 → Copilot → 向量索引**。当前版本仅支持 **OpenAI 兼容的 `/v1/embeddings`** 接口（覆盖 OpenAI、Qwen/DashScope、Ollama、Jina 等绝大多数提供商）。

---

## 配置项

| 项 | 说明 |
| --- | --- |
| 启用索引 | 总开关 |
| Embedding 模型 | 如 `text-embedding-3-small`、Qwen `text-embedding-v4` 等 |
| 向量维度（Dim） | **必须与模型实际输出维度一致**，详见下文「维度对齐」 |
| API Key | 访问密钥；本地服务（如 Ollama）可留空 |
| 自定义 API 地址 | 填服务的 **API 根地址**（如 `https://api.openai.com/v1`），系统会自动补 `/embeddings`，**请勿手动加** |
| 单批条数 | 重建索引时单次请求向量化的文本条数。部分提供商限制 64 或 25 条，超限会报错；填 `0` 用默认值 64 |

---

## 维度对齐（最容易踩的坑）

向量索引表在创建时会把维度**写死**成你填的 `Dim`，所以**模型实际返回的维度必须等于 `Dim`**，否则写入会报错：

```
Dimension mismatch ... Expected 1024 dimensions but received 2048
```

它的意思是：索引表按 `1024` 维建，但模型这次给了 `2048` 维。怎么填对：

- **支持自定义维度的模型**（OpenAI `text-embedding-3` 系、Qwen `text-embedding-v4` 等）：`Dim` 可在模型支持的档位里任选（如 `1024`、`1536`），Ech0 会请求该维度。
- **不支持自定义维度的模型**：`Dim` **必须填模型的原生维度**（查厂商文档，如 `1536`）。

::: tip 改了模型或维度怎么办
更换 **Embedding 模型** 或 **Dim** 后，旧索引即失效。Ech0 会提示「重建向量索引」——按提示**重建一次**，用新模型/维度给全部历史 Echo 重新生成向量即可。
:::

---

## 首次启用：重建索引

启用前发布的 Echo 还没有向量。配置好后，到「向量索引」里点 **重建索引（开始重建）**，为全部历史 Echo 补齐向量：

- 过程显示进度（已索引 / 总数 / 失败），可随时**取消**。
- 之后**新增 / 编辑 / 删除** Echo 会自动增量维护索引，无需再手动重建。
- 仅在**更换模型或维度**后才需要再重建一次。

---

## 费用提示

每条 Echo 入库、以及每次问答检索，都会调用一次 Embedding API，可能产生费用。请按需选择模型与提供商，并遵守其条款。
