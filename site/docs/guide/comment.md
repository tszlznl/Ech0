title: 评论系统

---

Ech0 使用**内置评论系统**，无需接入 Twikoo、Waline 等第三方评论托管。

---

## 启用与入口

在 **系统设置 → 评论设置**（或面板中的评论相关入口）中：

- 打开**全局评论开关**。
- 按需开启**先发后审**（访客评论需管理员通过后才公开展示）。
- 按需开启**验证码**（降低机器灌水风险；具体实现以当前版本为准）。

评论的**审核、删除、批量操作**一般在 **管理后台 → 评论管理** 中完成。

---

## 评论从哪来

常见类型包括：

- **访客评论**：未登录访客留言，通常需填写昵称，部分场景需邮箱；是否必须审核由设置决定。
- **已登录用户**：行为与权限受角色策略约束（以界面为准）。
- **集成接口**：自动化或上游系统可通过专用 API 写入评论（见下文「第三方集成」），来源会在后台可区分。

单条评论长度、富文本是否允许等以**当前版本与 OpenAPI** 为准。

---

## 审核与状态

评论会经历类似「待审核 → 通过 / 拒绝」的流程（具体状态名以界面为准）。管理员可以：

- 按状态、关键词筛选。
- 单条或批量通过、拒绝、删除。
- 将优质评论标为「热评」等（若版本提供）。

---

## 防灌水（思路）

实例可能组合使用：表单校验、频率限制、重复内容检测、验证码等。开关与阈值在**评论设置**里配置；需要精确参数时请查实例 **`/swagger/index.html`**。

---

## 第三方集成（AI / 自动化发评）

若要从自己的脚本、n8n、上层 AI **代为发评论**，可使用仓库 README 中说明的集成接口（路径形如 `POST /api/comments/integration`）：使用在 **访问令牌** 中创建的、具备 **`comment:write`** 等所需权限且 **`audience` 为 `integration`** 的令牌，在请求头携带 `Authorization: Bearer <token>`。

**不要**把该令牌提交到公开仓库。完整请求体、限频与字段说明以你实例上的 **OpenAPI（/swagger）** 为准。

---

## 与 Webhook

评论创建、状态变更、删除等可能触发 [事件推送](/docs/guide/webhook) 中的 `comment.*` 类事件，便于同步到 Slack、自建服务等。
